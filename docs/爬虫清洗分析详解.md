## 2. 模块一：爬虫 crawler.py

### 2.1 概述

- **文件位置**：`dujiangyan_weather/app/crawler.py`（567 行）
- **数据源**：[2345 天气网](https://tianqi.2345.com)，城市编码 `60407`（都江堰）
- **依赖**：`requests`、`BeautifulSoup4`、`re`
- **输出**：`WeatherData` 表（历史天气）、`ForecastData` 表（40 天预报）

### 2.2 爬虫架构

```
crawl_last_12_months()          ← 用户点击"一键爬取"
  │
  ├─ for 每个月:
  │    ├─ fetch_month(y, m)     ← 发送 HTTP 请求 + 解析 HTML
  │    ├─ parse_and_save()      ← 数据清洗 + 入库
  │    └─ time.sleep(2)         ← 反爬间隔
  │
  └─ 返回全部原始数据

fetch_forecast()                ← 用户点击"爬取预报"
  │
  ├─ 请求 40 天预报页面
  ├─ 正则提取内嵌 JSON
  ├─ 括号计数法定位完整数组
  └─ 逐条入库 ForecastData
```

### 2.3 历史天气爬取详解

#### URL 构造

请求 2345 天气网内部 API：

```
GET https://tianqi.2345.com/Pc/GetHistory
  ?areaInfo[areaId]=60407
  &areaInfo[areaType]=2
  &date[year]=2025
  &date[month]=6
```

方括号 `[` `]` 需要百分号编码为 `%5B` `%5D`（RFC 3986）：
```
areaInfo%5BareaId%5D=60407
```

#### API 返回格式

```json
{
  "data": "<table><tr><td>2025-06-01 周日</td><td>25°</td><td>18°</td><td>多云~晴</td><td>东南风 2级</td><td>37 优</td></tr>...</table>"
}
```

`data` 字段是一段 HTML 表格，每天一行 `<tr>`，每行 6 列 `<td>`。

#### 表格列映射

| `<td>` 索引 | 数据 | 原始示例 | 清洗后 |
|------------|------|---------|--------|
| 0 | 日期 | `"2025-06-01 周六"` | `"2025-06-01"` |
| 1 | 最高温 | `"25°"` / `"25℃"` | `25.0` |
| 2 | 最低温 | `"18°"` | `18.0` |
| 3 | 天气 | `"多云~晴"` / `"小雨转阴"` | 保留原文 |
| 4 | 风向风力 | `"东南风 2级"` | `direction="东南风"` `level="2级"` |
| 5 | 湿度 | `"37 优"` / `"-"` | `37.0` 或 `None` |

#### 温度清洗正则

```python
re.sub(r'[^\d.-]', '', "25°")   # → "25"
re.sub(r'[^\d.-]', '', "25.5℃") # → "25.5"
```

- `[^\d.-]` 匹配**非**数字、非小数点、非负号的字符，全部删掉
- 保留 `-` 是为了支持负温度（如 `-5℃`），保留 `.` 是为了支持小数

#### 湿度清洗

湿度可能有两种格式：
- `"37 优"` → 只取数字部分 `37`
- `"-"` → 该天无湿度数据，跳过

#### 风向风力拆分

原始字段 `"东南风 2级"` 按空格 split：
```python
["东南风", "2级"]
  → wind_direction = "东南风"
  → wind_level      = "2级"
```

### 2.4 40 天预报爬取详解

#### 与历史天气爬虫的区别

| 对比维度 | 历史天气 | 40 天预报 |
|---------|---------|----------|
| 数据源 | API 接口 | HTML 页面 |
| 返回格式 | JSON → `data` 是 HTML 表格 | HTML → `<script>` 内嵌 JSON |
| 解析方式 | BeautifulSoup 解析 `<tr>` | 正则提取 + JSON 解析 |
| 数据量 | 每请求 1 个月（~30 条） | 每请求 40 条 |
| 去重键 | `date` | `date` |

#### JSON 提取的核心难点：括号计数法

页面源码中 JSON 可能包含嵌套结构：

```javascript
"data":[
  {...},
  {...},
  // ... 40 条
]
```

如果用简单的非贪婪匹配 `.*?`，会在遇到第一个 `]` 时就停止，导致 JSON 被截断（因为 JSON 对象内的字符串也可能包含 `]`）。

**解决方案：括号计数法**

```python
bracket_count = 0
for i, c in enumerate(raw):
    if c == '[':
        bracket_count += 1   # 遇到左括号 +1
    elif c == ']':
        bracket_count -= 1   # 遇到右括号 -1
        if bracket_count == 0:
            end_idx = i + 1  # 计数归零 = 找到配对的右括号
            break
```

原理与传统括号匹配算法一致：
- 初始化计数器为 0
- 遇到 `[` 计数+1，遇到 `]` 计数-1
- 当计数器首次归零时，找到了与第一个 `[` 配对的最外层 `]`

### 2.5 反爬策略（4 层防护）

| 层次 | 措施 | 位置 |
|------|------|------|
| ① | 完整浏览器请求头（UA + Referer + X-Requested-With + Accept-Language） | `HEADERS` 常量 |
| ② | 每月请求间隔 2 秒，模拟人类浏览节奏 | `crawl_last_12_months()` 中 `time.sleep(2)` |
| ③ | 每个请求最多尝试 2 次（首次 + 1 次重试），重试前等待 2 秒 | `fetch_month()` 中 `max_attempts=2` |
| ④ | 单月失败不中断全局，跳过继续下一个 | 异常处理返回 `[]`，外层继续循环 |

### 2.6 去重策略

```python
WeatherData.objects.update_or_create(
    date=date_obj,       # 以 date 为唯一键查找
    defaults={...}       # 存在则更新，不存在则创建
)
```

- 重复爬取同一月份不会产生重复行
- 数据源修正后（如之前温度有误），再次爬取会自动覆盖旧值

### 2.7 CrawlTask 状态机

```
pending ──→ running ──→ success  (records_count=N, completed_at=now)
                    └─→ failed   (error_message=..., completed_at=now)
```

作用：前端可以查询"上次爬取 X 月成功/失败了吗"，给用户反馈。

## 3. 模块二：数据清洗 data_cleaner.py

### 3.1 概述

- **文件位置**：`dujiangyan_weather/app/data_cleaner.py`（191 行）
- **依赖**：`pandas`、`numpy`
- **输入**：`WeatherData` 表中的原始数据
- **输出**：清洗后的 `WeatherData` 表（原地更新）

### 3.2 清洗流程（6 步）

```
run_cleaning()
  │
  ├─ 步骤1: get_dataframe()          从数据库读取 → DataFrame
  ├─ 步骤2: standardize_and_dedup()  日期标准化 + 去重
  ├─ 步骤3: clean_outliers()         异常值处理
  ├─ 步骤4: clean_missing_values()   缺失值填充
  ├─ 步骤5: add_month_column()       新增月份列
  ├─ 步骤6: compute_monthly_stats()  按月聚合统计（仅日志）
  └─ 步骤7: save_cleaned()           回写数据库
```

### 3.3 各步骤详解

#### 步骤 1：读取数据 `get_dataframe()`

```python
df = pd.DataFrame(WeatherData.objects.all().values())
df['date'] = pd.to_datetime(df['date'], errors='coerce')
df = df.sort_values('date').reset_index(drop=True)
```

- `errors='coerce'`：无法解析的日期变成 `NaT`，避免因一条坏数据中断整个流程
- 按日期升序排列，为后续的前向填充做准备

#### 步骤 2：去重 `standardize_and_dedup()`

```python
df = df.drop_duplicates(subset=['date'], keep='first')
```

- 以 `date` 为去重键
- `keep='first'`：同一天保留第一条记录
- 去重前先 `dropna(subset=['date'])` 移除无效日期行

#### 步骤 3：异常值处理 `clean_outliers()`

**阈值设定**：
- 最高温 > 50°C → 异常（都江堰历史上从未超过 40°C）
- 最低温 < -30°C → 异常（都江堰地处四川盆地，极少低于 -5°C）

**处理流程**：
```
检测异常值 → 标记为 NaN → 用该月均值替换 → 整月缺失用全局中位数兜底
```

```python
df.loc[(df[col] > 50) | (df[col] < -30), col] = np.nan           # 标记
df[col] = df.groupby('month')[col].transform(lambda x: x.fillna(x.mean()))  # 月均值填充
df[col] = df[col].fillna(df[col].median())                        # 全局中位数兜底
```

#### 步骤 4：缺失值填充 `clean_missing_values()`

**填充策略（两级兜底）**：

```
优先：前向填充（用前一天同字段的值）
  ↓ 如果开头就是 NaN（无前一天）
兜底：该列中位数
```

```python
df[col] = df[col].ffill()                     # 前向填充
df[col] = df[col].fillna(df[col].median())    # 中位数兜底
```

选择 `ffill` 而非均值的原因：天气数据具有时间连续性，前一天的温度通常比全月均值更接近真实值。

#### 步骤 5：新增月份列

```python
df['month'] = df['date'].dt.to_period('M').astype(str)
# 示例：date=2025-06-15 → month="2025-06"
```

#### 步骤 6：按月统计 `compute_monthly_stats()`

聚合维度：
- `max_temp`：月均最高温
- `min_temp`：月均最低温
- `weather_desc`：天气分布（`{"晴天": 12, "多云": 8, "雨天": 5, ...}`)
- `rainy_days`：降雨天数（天气描述含"雨"字的行数）

注意：这里的统计仅输出到日志，不直接入库。真正的入库统计由 `analyzer.py` 完成。

#### 步骤 7：回写数据库 `save_cleaned()`

```python
for _, row in df.iterrows():
    obj = WeatherData.objects.get(id=row['id'])
    obj.max_temp = round(row['max_temp'], 2)   # 保留 2 位小数
    obj.min_temp = round(row['min_temp'], 2)
    obj.humidity  = round(row['humidity'], 2)
    obj.save()
```

- 逐行更新，非批量（因为是清洗场景，数据量不会特别大）
- `round(..., 2)` 保证数据库内数据精度一致

## 4. 模块三：数据分析 analyzer.py

### 4.1 概述

- **文件位置**：`dujiangyan_weather/app/analyzer.py`（606 行）
- **依赖**：`pandas`、`numpy`、`django.db.models`（聚合函数）
- **输入**：`WeatherData` 表（清洗后） + `ForecastData` 表
- **输出**：`MonthlyStats` 表 + `ClothingAdvice` 表

### 4.2 分析架构

```
analyze_all()
  │
  └─ generate_monthly_stats(year, month)
       │
       ├─ 1. 从 WeatherData 读取 → pandas DataFrame
       ├─ 2. 按 (year, month) 分组
       ├─ 3. 每个月份计算：
       │     ├─ 温度统计（均最高/均最低/极最高/极最低）
       │     ├─ 降雨天数
       │     ├─ 平均湿度
       │     ├─ 天气分布（JSON）
       │     └─ 5 项气候评分（见 4.3）
       ├─ 4. 写入 MonthlyStats 表
       └─ 5. 生成智能建议
              ├─ 优先：AI 引擎（ai_advisor.py → DeepSeek API）
              └─ 兜底：规则引擎（generate_comprehensive_advice）
```

### 4.3 5 项气候评分算法

每项评分统一归一化到 **0–100** 分。

#### ① 温度舒适度 `_calc_temp_comfort_score()`

```
评分 = (舒适天数 / 总天数) × 100

舒适定义：最高温 ∈ [18, 30] 且 最低温 ∈ [10, 25]
```

- 18~30°C 是人体最舒适的室外温度区间
- 夜间不低于 10°C 且不高于 25°C 保证睡眠质量

#### ② 湿度适宜度 `_calc_humidity_comfort_score()`

```
评分 = max(0, 100 - |实际湿度 - 50| × 2)

偏离理想值 50% 每 1%，扣 2 分
```

- 50% 是公认的人体舒适湿度
- 都江堰常年湿度较高（70%+），分数通常偏低

#### ③ 日照充足度 `_calc_sunlight_score()`

```
评分 = ((晴天 + 多云天数) / 总天数) × 100
```

- 基于 `weather_type` 字段（由 `WeatherData.save()` 自动推断）
- 阴天、雨天不计入

#### ④ 空气质量 `_calc_air_quality_score()`

```
评分 = 晴天比例得分(60%) + 降雨适中度得分(40%)

晴天比例得分 = (晴天数 / 总天数) × 60
降雨适中度   = max(0, 100 - |月降雨天数 - 6| × 12) × 0.4
```

- 因无实际 AQI 数据，用天气类型作为间接指标
- 晴天多 → 大气扩散条件好
- 每月 5~8 天降雨最佳 → 能净化空气又不至于太潮湿

#### ⑤ 降水适中度 `_calc_precipitation_score()`

```
评分 = max(0, 100 - |月降雨天数 - 6| × 12)
```

- 6 天/月 为理想降雨频率（适度湿润但不影响出行）
- 偏离最佳值越大，分数越低

### 4.4 智能建议：双引擎设计

```
generate_monthly_stats()
  │
  ├─ 尝试 AI 引擎
  │    └─ generate_ai_advice(month_str)
  │         ├─ 构建天气上下文（近30天 + 未来7天预报 + 月度统计）
  │         ├─ 发送给 DeepSeek v4-pro
  │         └─ 返回 JSON → 写入 ClothingAdvice
  │
  └─ AI 失败 → 回退规则引擎
       └─ generate_comprehensive_advice(year, month)
            ├─ 数据准备：近60天历史 + 未来14天预报
            ├─ 5 类 if/else 规则判断
            └─ 写入 ClothingAdvice
```

#### AI 引擎（优先）

```python
# ai_advisor.py
context = _build_weather_context(month_str)
# context 包含：
#   近30天实际数据（日均温、极端温、降雨天数、最近7天明细）
#   未来7天预报
#   当月月度统计

prompt = f"""
你是一个天气生活顾问。根据以下都江堰天气数据，生成 5 类生活建议。
{context}
严格按 JSON 格式输出...
"""

resp = _call_ai(messages=[...], temperature=0.5)
return json.loads(resp)
```

- 模型：DeepSeek v4-pro（兼容 OpenAI 格式）
- 温度参数 0.5（偏低，保证输出稳定）
- max_tokens 8000
- 响应中优先取 `content`，为空时用 `reasoning_content` 兜底（适配推理模型）

#### 规则引擎（兜底）

当 AI 调用失败（网络超时、API 欠费、返回格式异常）时自动回退。

**5 类建议的规则逻辑**：

| 分类 | 判断维度 | 阈值示例 |
|------|---------|---------|
|  穿衣 | 气温区间 + 温差 + 湿度 | >32°C→短袖，12~20°C→薄外套，<5°C→羽绒服 |
|  出行 | 降雨概率 + 极端天气 | ≥5天雨→带雨具，>35°C→防暑，<0°C→防滑 |
|  运动 | 温度 + 降雨 | 15~28°C 少雨→户外运动，>33°C→游泳/室内 |
|  健康 | 湿度 + 温差 | 湿度>80%→防霉通风，温差>12°C→防感冒 |
| ️ 预警 | 极端温 + 暴雨 | ≥38°C→danger，≥35°C→warning，暴雨≥2天→warning |

### 4.5 其他分析函数

#### 天气分布 `get_weather_distribution()`

```
输入：year, month（可选）
输出：[{name: "晴天", value: 15, color: "#D4A373"}, ...]

颜色映射：
  晴 → #D4A373 (暖黄)
  多云 → #A8A29E (浅灰)
  阴 → #78716C (深灰)
  雨 → #7FA3C1 (蓝色)
  雪 → #B8C5D6 (淡蓝)
```

按天气描述子串匹配颜色（含"雨"字即用蓝色，含"雪"字即用淡蓝），确保复合天气（如"多云转晴"）也能正确配色。

#### 热力图 `get_heatmap_data()`

对每个月计算 5 个分位值：

```
row 0: min      (当月最低温)
row 1: Q25      (25% 分位)
row 2: median   (中位数)
row 3: Q75      (75% 分位)
row 4: max      (当月最高温)
```

前端用这 5 行数据渲染温度热力图（12 列 × 5 行），颜色深浅代表温度高低。

#### 气候评分 `get_climate_scores()`

从 `MonthlyStats` 表读取 5 项评分，多个月份时取平均值，返回雷达图所需数据。

## 5. 三个模块的协作关系

### 5.1 数据流依赖

```
crawler.py          data_cleaner.py       analyzer.py
    │                     │                    │
    │  写入原始数据        │                    │
    ├──────────────────→  │                    │
    │  WeatherData         │                    │
    │                     │  读取 + 清洗 + 回写 │
    │                     ├──────────────────→  │
    │                     │  WeatherData(干净)  │
    │                     │                    │  读取 + 分析
    │                     │                    ├──────────→ MonthlyStats
    │                     │                    │            ClothingAdvice
    │                     │                    │
    │  写入预报数据        │                    │
    ├─────────────────────────────────────────→  │
    │  ForecastData                              │  读取预报数据
    │                                            ├──────────→ 综合建议
```

### 5.2 调用入口

| 触发方式 | 调用的函数 | 对应 API |
|---------|-----------|---------|
| 前端"一键爬取"按钮 | `crawler.crawl_last_12_months()` → `parse_and_save()` | `POST /api/weather/crawl/` |
| 前端"爬取预报"按钮 | `crawler.fetch_forecast()` | `POST /api/weather/forecast/fetch/` |
| 前端"一键分析"按钮 | `analyzer.analyze_all()` → `generate_monthly_stats()` | `POST /api/weather/analyze/` |
| 命令行 | `python manage.py run_pipeline` | 先 `run_cleaning()` 再 `analyze_all()` |

### 5.3 容错设计

```
爬虫层：
  单月失败 → 标记 CrawlTask failed → 继续下一个月 ✓

清洗层：
  单条异常 → ffill/中位数兜底 → 不影响其他行 ✓

分析层：
  AI 调用失败 → 自动回退规则引擎 → 保证有结果 ✓
  单月无数据 → 跳过该月 → 其他月正常分析 ✓
```

每一层都设计为**局部失败不影响全局**，保证系统在部分数据缺失时仍能正常运行。
